  01    SECURITY IMPROVEMENTS
1. Security Improvements
The current app has solid foundations but several attack surfaces that must be hardened before exposing it to the internet. Each fix below explains the vulnerability, the risk, and the exact implementation.
1.1  Login Throttling — Add Per-Username Lock
Current state: The throttle tracks by IP address only. An attacker who rotates IPs (VPN hops, tor exit nodes, mobile data) can bypass the limit entirely and brute-force usernames.
Fix — track by username hash, not just IP:
# In app.py — add a username-keyed attempt counter alongside the IP counter
failed_attempts_by_user: dict[str, int] = {}
def check_rate_limit(username: str, ip: str):
    key_user = hashlib.sha256(username.encode()).hexdigest()[:16]
    if failed_attempts_by_user.get(key_user, 0) &gt;= 10:
        raise HTTPException(429, &apos;Too many attempts for this username&apos;)
    if failed_attempts_by_ip.get(ip, 0) &gt;= 20:
        raise HTTPException(429, &apos;Too many attempts from this IP&apos;)
!
Why it matters
Without per-username locking a single attacker on a VPN can try thousands of passwords. With both IP and username tracking, even rotating IPs cannot brute force a specific account.
1.2  SameSite=Strict on Session Cookie
The current session cookie does not explicitly set SameSite=Strict. This exposes the app to CSRF attacks where a malicious page causes the victim&apos;s browser to silently send authenticated requests.
# In app.py — update the session middleware config
app.add_middleware(SessionMiddleware,
    secret_key=SECRET_KEY,
    session_cookie=&apos;session&apos;,
    same_site=&apos;strict&apos;,     # &lt;- add this
    https_only=True,        # &lt;- add this (Cloudflare handles TLS)
    max_age=86400)
1.3  Secrets Not in Project Root
Both .env and passkeys.local.json sit in the project root. If a developer accidentally pushes to GitHub or shares a zip, credentials and passkey private material leak.
Create a secrets/ directory outside the project root: C:\Users\&lt;name&gt;\AppData\Local\FamilyCCTV\secrets\
Point the app to it via a single SECRETS_DIR env var set at OS level (not in .env)
Add .env, passkeys.local.json, and *.key to .gitignore with a pre-commit hook that blocks the push if they exist
1.4  Camera Health Probe Endpoint
If OpenCV fails to open the camera (another app holds the device, driver crash), the error propagates silently into the WebRTC negotiation. The viewer sees a black screen with no explanation.
# New route in app.py
@app.get(&apos;/health/camera&apos;)
async def camera_health():
    cap = cv2.VideoCapture(CAMERA_INDEX)
    ok = cap.isOpened()
    cap.release()
    return {&apos;status&apos;: &apos;ok&apos; if ok else &apos;unavailable&apos;,
            &apos;index&apos;: CAMERA_INDEX}
The viewer page should call this before initiating WebRTC and show a friendly inline message (&quot;Camera not available — check that no other app is using it&quot;) instead of a stalled spinner.
1.5  WebRTC Fallback Chain with UI Feedback
When TURN fails, the viewer silently stalls. Users on mobile networks (CGNAT, symmetric NAT) experience this frequently. The fix is an explicit escalation chain with status messages visible in the UI.
Attempt 1: Direct P2P (fastest, fails behind symmetric NAT)
Attempt 2: STUN-assisted (works on most home networks)
Attempt 3: TURN relay via Cloudflare Realtime
Fallback: Serve MJPEG stream from /video endpoint (always works, higher latency)
The viewer JS should update a status label at each step so the user knows what is happening rather than seeing a frozen camera icon.
1.6  Session Expiry Notification via Server-Sent Events
When the watchdog kills a stale WebRTC session, the browser gets no notification. The video freezes and the user has no idea whether to refresh or wait. Add an SSE channel:
@app.get(&apos;/events&apos;)
async def sse_events(request: Request):
    async def generator():
        while True:
            event = await session_event_queue.get()
            yield f&apos;data: {json.dumps(event)}\n\n&apos;
    return StreamingResponse(generator(),
        media_type=&apos;text/event-stream&apos;)
The viewer subscribes to /events and shows a banner (&quot;Session expired — tap to reconnect&quot;) instead of a frozen screen.
1.7  Video Message Size Enforcement + Auto-Cleanup
The received_messages/ folder has no size cap and no cleanup policy. A viewer who sends many messages will silently fill the laptop disk.
Enforce server-side max upload size (current .env.example has a limit var — make it strict in the route handler)
On each new upload, delete messages older than 7 days
Keep a max of 20 files — delete oldest if exceeded
Return a 413 with a user-friendly JSON body, not a generic 500
1.8  Input Validation on All Routes
FastAPI&apos;s Pydantic models should validate every query param and JSON body field. Review all routes and ensure that string fields have max_length constraints, integers have ge/le bounds, and file uploads check MIME type against an allowlist (video/mp4, video/webm only).
  02    ARCHITECTURE IMPROVEMENTS
2. Architecture Improvements
2.1  Async Camera Frame Buffer
The current CameraManager reads frames synchronously. Under high CPU load (screenshot capture, video transcoding) the read loop can block and cause dropped frames or WebRTC jitter.
Solution: run camera reads in a thread pool and push frames into an asyncio.Queue.
import asyncio, concurrent.futures
class CameraManager:
    def __init__(self):
        self._queue: asyncio.Queue = asyncio.Queue(maxsize=5)
        self._executor = concurrent.futures.ThreadPoolExecutor(max_workers=1)
    async def _read_loop(self):
        loop = asyncio.get_event_loop()
        while self._running:
            frame = await loop.run_in_executor(self._executor, self._cap.read)
            if not self._queue.full():
                await self._queue.put(frame)
2.2  Config Dataclass Instead of Bare ENV Reads
Currently os.getenv() calls are scattered throughout app.py. This makes it easy to miss a missing var at startup and hard to see all config in one place.

2.3  Structured Logging
Replace print() statements with Python&apos;s logging module configured with structured JSON output. This makes log filtering trivial (grep for session_id, event_type) and enables future log shipping to a file or remote sink.
import logging, json

    def format(self, record):
        return json.dumps({&apos;ts&apos;: self.formatTime(record),

2.4  Video Message Transcoding on Receipt
Uploaded video messages are saved as-is. If the viewer&apos;s browser records in a codec the laptop&apos;s media player does not support (VP9 webm on Windows Media Player), playback silently fails.
Transcode on receipt using ffmpeg:
import subprocess
def transcode(src: Path, dst: Path):
    subprocess.run([
        &apos;ffmpeg&apos;, &apos;-i&apos;, str(src),
        &apos;-c:v&apos;, &apos;libx264&apos;, &apos;-c:a&apos;, &apos;aac&apos;,
        &apos;-movflags&apos;, &apos;+faststart&apos;,  # enables progressive web playback
        str(dst)], check=True)

Feature 1 — Motion Alert &amp; Push Notification
What it does: When no viewer is actively watching, a background thread performs frame differencing. If pixel change exceeds a threshold, the server sends a Web Push notification to the viewer&apos;s phone.
Why: Right now the app is entirely pull-based. The viewer must open the app and manually check. Motion detection converts the app into a genuine security tool — you get notified without doing anything.
Implementation:
Install pywebpush: pip install pywebpush
Add POST /subscribe endpoint that stores the browser&apos;s PushSubscription object
In CameraManager, run a background diff loop when viewer_count == 0
Compare grayscale frames: cv2.absdiff() followed by cv2.threshold()
If changed_pixels / total_pixels &gt; 0.02 (2% threshold), call send_push()
Store VAPID keys in .env (generate once with pywebpush)
Add cooldown: do not send more than one alert per 60 seconds
ℹ
Sensitivity tip
Start with a 2% pixel change threshold. Expose this as a slider in the viewer settings tab so the user can tune it to their environment (a fan blowing curtains vs a person walking).
Feature 2 — Two-Way Audio
What it does: Adds a microphone track from the viewer&apos;s browser to the WebRTC connection. The laptop plays the viewer&apos;s voice through its speakers in real time.
Why: Video-only monitoring is passive. Two-way audio lets a parent call out to a child, reassure a pet, or have a conversation. aiortc already handles this — it is largely a JS-side change.
Implementation:
In viewer.html: navigator.mediaDevices.getUserMedia({ audio: true, video: false })
Add the audio track to the existing RTCPeerConnection before the offer
On the server (app.py), the incoming audio track is already received by aiortc
Route the audio to the system speaker using pyaudio: pip install pyaudio
Add a mute/unmute button in the viewer UI (mic icon, bottom sheet)
Add a speaker volume indicator on the viewer so the user knows audio is live
Feature 3 — On-Demand 30-Second Clip Save
What it does: The server maintains a rolling 30-second frame buffer in memory. When the viewer taps &apos;Save Clip&apos;, the buffer is flushed to an MP4 file on disk.
Why: Users want to save evidence of something they just saw without committing to continuous recording (which fills disks and raises privacy concerns). The ring-buffer approach gives the best of both.
Implementation:
Use a collections.deque(maxlen=900) at 30fps — this holds exactly 30 seconds
Each entry is an encoded JPEG or raw numpy frame from CameraManager
When save is triggered: flush deque, write with cv2.VideoWriter to MP4
Store in saved_clips/ directory (separate from received_messages/)
Return the filename and a download link in the JSON response
Add the clip to the Clips tab in the viewer UI
Feature 4 — Battery &amp; System Status Panel
What it does: A /status/system endpoint returns laptop battery %, CPU load, RAM usage, disk free space, and network interface status.
Why: If the viewer starts a live session and the laptop is at 5% battery with no charger, the session will die mid-stream. Showing system health upfront lets the viewer make an informed decision.
Implementation:
import psutil  # pip install psutil

Show these as four metric cards on the Idle screen (battery icon, wifi icon, CPU icon, disk icon) that refresh every 30 seconds.
Feature 5 — Night Vision / Low-Light Mode
What it does: A toggle in the viewer switches the camera capture to a low-light preset: slower shutter (higher exposure), boosted ISO/gain, and optional grayscale mode.
Why: Most of the use-cases for family monitoring (checking on a sleeping child, a pet at night) happen in low-light. Default camera settings produce dark, noisy frames.
Implementation:

Feature 12 — Digital PTZ (Pan/Tilt/Zoom)
What it does: Software-based crop and zoom of the camera frame. The viewer can pinch-to-zoom on mobile or use +/- buttons to zoom into a region of the frame.
Why: Physical cameras cannot pan or tilt, but a software crop can simulate it. If the camera covers a room, the viewer can zoom in on a specific area without physically adjusting the camera.
def crop_and_zoom(frame, cx, cy, zoom):
    h, w = frame.shape[:2]
    crop_w, crop_h = int(w / zoom), int(h / zoom)
    x1 = max(0, int(cx - crop_w/2))
    y1 = max(0, int(cy - crop_h/2))
    cropped = frame[y1:y1+crop_h, x1:x1+crop_w]
    return cv2.resize(cropped, (w, h))

  04    UI/UX REDESIGN — ANDROID 9:16
4. UI/UX Redesign Guide
The current viewer.html is a desktop-first layout. On a 9:16 Android screen in portrait mode, controls are small, the video does not fill the screen, and the action buttons are scattered. This section defines the complete mobile-first redesign.
4.1  Core Design Principles
1
Thumb zone first
All interactive controls must sit in the bottom 40% of the screen (below the midpoint). Nothing important above the midpoint — the user&apos;s thumb cannot comfortably reach there in one-handed use.
2
Video fills everything
The camera feed should fill the entire viewport behind all other UI elements. Controls overlay the video with semi-transparent backgrounds, never block it with opaque panels.
3
Dark theme is default
Security camera apps are checked at night, in dark rooms, during anxiety. A dark UI (#0a0a0a base) is gentler on the eyes and feels more appropriate for the use case.
4
Minimal tap targets — 48px minimum
All buttons must be at least 48x48px per Android Material guidelines. The current layout has several buttons that are far too small for reliable tapping.
5
One primary action per screen state
In Live mode: the primary action is Stop. In Idle mode: the primary action is Start. Do not present five equal-weight buttons. Visual hierarchy must be immediately obvious.
4.2  Screen Architecture — Five States
State 1: Login screen
Full-screen dark background (#090909)
Centered app icon (home-shield, 52px, green-tinted card)
App name in 20px medium weight, subtitle in 12px muted
Username and password fields with floating labels (not placeholder text)
Primary &apos;Sign in&apos; button: full width, green tinted background, green text
Divider line with &apos;or&apos; label
Passkey button: full width, outlined, fingerprint icon + &apos;Use passkey&apos;
No logo image — icon only. No decorative background patterns.
State 2: Idle / Connect screen
Top half: camera location name, connection status indicator, last-seen time
Center: large circular start button (80px, green ring, play icon inside)
Below button: 2x2 grid of system status cards (battery, wifi, CPU, disk)
Bottom navigation bar: Live | Clips | Alerts | Settings tabs
Status cards auto-refresh every 30 seconds via polling /status/system
State 3: Live view screen
Camera feed fills 100% of viewport, portrait crop

5.2  Startup Time
The server currently imports OpenCV, aiortc, mss, and webauthn at startup. On a slow laptop this can take 3-5 seconds before the first request is served. Defer heavy imports:
# Lazy import pattern
_cv2 = None
def get_cv2():
    global _cv2
    if _cv2 is None:
        import cv2 as _cv2
    return _cv2
This brings the server to &apos;ready to accept login&apos; state within 0.5 seconds. The heavier modules load the first time a camera session is requested.
5.3  Windows Service Installation
Currently the app must be manually started via PowerShell. Add a one-command install as a Windows service so it starts automatically at boot:
# Install NSSM (Non-Sucking Service Manager) then:
nssm install FamilyCCTV python app.py
nssm set FamilyCCTV AppDirectory C:\Users\&lt;user&gt;\family_cctv_project
nssm set FamilyCCTV AppEnvironmentExtra PATH=%PATH%
nssm start FamilyCCTV
Add Uninstall-FamilyCctvService.ps1 to the scripts/ directory.
5.4  Automated Cloudflare Tunnel Restart
The current Start-CloudflareQuickTunnel.ps1 launches the tunnel but does not restart it if it crashes. Add a watchdog loop:
# In Start-CloudflareQuickTunnel.ps1
while ($true) {
    $proc = Start-Process cloudflared -ArgumentList &apos;tunnel --url http://localhost:8000&apos; -PassThru
    $proc.WaitForExit()
    Write-Host &quot;Tunnel exited (code $($proc.ExitCode)) — restarting in 5s&quot;
    Start-Sleep 5
}
